# Spec 766 — Runtime Recorder: shared-memory data-streaming (anchors + async worker)

**Status:** DRAFT (2026-06-15) — for refinement with the user, THEN build.
**Supersedes:** Spec 765 (the interim flat in-process ring / "Mittelweg" — it kept
heavy work on the emu thread and never reached the clean model; this is the
real design). 765's framebuffer-is-derivable insight carries over.
**Pairs with / reuses:** 746 (live-trace+scrub charter — the anchor+firehose
model this realizes), 726.B (the zero-alloc binary firehose + worker — the
pattern to copy), 707 (.c64re dump/undump format), 705/705.B (the capability),
612 (1541 port-fidelity — for the disk write-generation).

## 0. Why this exists (the user's words, ratified 2026-06-15)

BUG-049: always-on checkpoint capture made the live audio "kratzen" + dropped
fps to 43-47. Every fix was a patch on the wrong model — the root error was
doing **heavy work on the emulation thread every capture** (full framebuffer
copy, and 3× sha256 of the ~1 MiB EF cartridge image PER SECOND to decide
dedup). That is "evaluating the recorder at runtime", which the recorder must
NEVER do.

The user's model, simple and clean: **no GC, no node-runtime cruft — plain
memcpy then async. Data-streaming.**

## 1. The model — three actors, strictly separated

```
  TICK LOOP (emu thread)          RUNTIME owns the 128 MiB           ASYNC WORKER (worker_thread)
  ───────────────────────         ─────────────────────────         ───────────────────────────
  fire-and-forget:                small SAB handoff ring             drains the handoff (read-cursor);
  memcpy (timestamp, RAM,   ───►  (loop writes, worker        ───►  does ALL heavy work:
  CPU/chip state, IO,             reads — both see the               • store full anchors (no diff)
  + medium ONLY on gen-change)    same bytes); + a worker-           • dedup the medium by generation
  → advance Atomics write-cursor  owned ~32 MiB anchor store         • index (cycle → offset)
  NO read, NO hash, NO GC.                                           • serve the UI/MCP API
```

**Invariants (non-negotiable — they are the whole point):**

- **The emu thread NEVER reads or evaluates the recorder.** It only `memcpy`s raw
  bytes into the shared ring and bumps an `Atomics` write-cursor. No sha256, no
  dedup, no diff, no JS object graph, no allocation on the hot path → no GC.
- **The 128 MiB belongs to the Runtime, not the tick loop.** The loop reaches it
  through one internal "deliver" API and forgets.
- **All heavy work is on the worker, async.** Delta computation, medium dedup,
  compaction, indexing, and the UI-facing evaluation API. A slow worker can
  never stall the emu thread (a backed-up ring drops oldest = a recorder gap,
  exactly like a ring eviction — never backpressure onto emulation).
- **The medium is delivered only when it changed**, gated by an O(1) generation
  number the loop reads cheaply (never hashed on the hot path).

This is the same shape the binary CPU firehose already runs in (726.B:
zero-alloc producer on the emu thread, worker writer/indexer). We are bringing
the checkpoint/anchor side onto that same pattern — it was the lone holdout
doing heavy work inline.

## 2. What the loop delivers (the record stream)

Framed records, appended round-robin into the shared ring. Frame header:
`[type:u8][len:u32][cycle:f64][frame:u32]`. Types:

- **ANCHOR** — a full machine state, every N (cadence ~0.5–1 s, configurable):
  - RAM (64 KiB) — raw memcpy.
  - CPU regs + CIA/SID/VIC/IEC/alarms **scalars**, packed flat (little-endian
    fields — the only "serialization", and it is tiny + mechanical, no object
    graph). **NOT** the framebuffer (derivable shadow, regenerated by re-sim on
    restore — Spec 765 §8 / 746 §1 "only the CPU writes RAM").
  - IO snapshot.
  - The current medium **generation numbers** (cart + disk), O(1).
- **MEDIUM** — the medium bytes (disk GCR image and/or cart .crt + flash),
  delivered **only when its generation changed** since the last delivery. Carries
  its generation so the worker keys dedup on it (no hashing on the hot path).

The loop computes NO deltas. It delivers full anchors (a 64 KiB memcpy is
microseconds) + medium-on-change. **The worker stores full anchors as-is** (RQ3:
NO delta encoding — at 0.5 s a ~32 MiB store already holds ~3-4 min, which is
plenty; delta-diffing would only trade worker CPU for history we don't need).
(Sub-anchor fine stepping is the separate CPU firehose / trace — §6, reused.)

## 3. Memory + threading

- **Small `SharedArrayBuffer` handoff ring** (a few seconds of raw anchors, e.g.
  ~2-4 MiB; sized in 766.1). Pre-allocated + paged-in ONCE at power-on (before the
  first frame + audio — Spec 765 §8 prewarm), so allocation never competes with
  the boot.
- **Worker-owned anchor store**, a normal `ArrayBuffer` (~32 MiB default,
  `recorderBudgetBytes?` per session — ~3-4 min at 0.5 s; full anchors, RQ3). NOT
  shared (only the worker + the UI-over-WS API touch it).
- **Ring discipline** via `Atomics`: a write-cursor (loop) and a read-cursor
  (worker) on the handoff. The loop `Atomics.store`s the write-cursor after each
  frame; the worker polls/`Atomics.wait`s and drains into its store.
  Overwrite-oldest in the handoff when the worker is behind (a recorder gap,
  logged — never block the loop).
- **No transferables, no postMessage of data** — the bytes live in the handoff
  SAB; only small control signals cross (cursors via Atomics). Zero per-frame
  allocation on the loop.
- **Worker = `worker_thread`** (same mechanism as `binary-log-worker.ts`). It owns
  the anchor store + the index + the query API.

## 4. The worker (all heavy work, async)

- **Store full anchors** round-robin in the ~32 MiB store (RQ3: no delta diff —
  ~3-4 min depth is plenty; the worker stays a dumb-fast copier+indexer, no CPU
  burned diffing). **Pure overwrite-oldest, no exceptions** (RQ4: no pin). The ring
  is a transient rolling window; durability is the explicit dump (§6), never
  automatic.
- **Dedup the medium** by generation (the loop only delivers it on change): store
  each distinct disk/cart/flash version once, referenced by anchors. This replaces
  the per-second sha256 — identity is the O(1) generation, not a content hash
  recomputed every second.
- **Index**: cycle/frame → stored-anchor offset, for random-access scrub.
- **Serve** the UI/MCP evaluation API: list anchors, restore-data for an anchor
  (its full bytes + the referenced medium), dump a `.c64re` from an anchor (§6),
  stats.

## 5. Medium — FEATURE COMPLETE (no deferral)

Both media get the same O(1) generation gate; the loop delivers the medium only
when its generation moved.

- **Cartridge:** `writableGeneration()` already exists (cartridge.ts — monotonic,
  bumped on flash write). The original .crt bytes are immutable (generation never
  moves for them → delivered once); the flash image moves only on EAPI writes.
- **Disk:** ADD `diskWriteGeneration(): number` to the vice1541 facade — an O(1)
  monotonic counter the BRIDGE maintains (bumped when `GCR_dirty_track` transitions
  nonzero / on writeback), read-only on the VICE core (Spec 612 PL-5: bridge
  bookkeeping lives in the facade, NOT inside the ported core; no core edit).
  The loop reads it O(1) and delivers the GCR image only when it moved.

Persistence (the lazy medium-RAM→filesystem writer, BUG-040 cart auto-persist +
disk writeback) stays as-is; it reads the same generation. Recorder and
persistence are independent readers of the same O(1) signal — neither hashes on
the hot path.

## 6. Integration with the existing surfaces (reuse, don't rebuild)

- **Scrub** (coarse, always-on): restore an anchor via the worker's restore-data
  API → the controller applies it (kernel.restore) → run one frame to repaint
  (the framebuffer is regenerated, Spec 765 §8). The existing WS `checkpoint/*`
  + MCP `runtime_checkpoint_*` (746.4) re-point at the worker API.
- **Explicit dump from the ring = the ONLY durability** (RQ4: no auto-pin). Dump a
  chosen anchor (or the current ring head) to a full-fidelity `.c64re` (707; the
  framebuffer is regenerated at dump time by a 1-frame re-sim so undump shows the
  screen + 707 stays green). Reachable from THREE gates, mirroring the trace
  control (746 OQ1): a **Monitor command**, an **MCP tool** (LLM), and a **UI
  button** (human). Never automatic.
- **Fine trace** (on demand): the CPU_STEP firehose (726.B) + the "⏺ Trace"
  button (746.9) — UNCHANGED. The user's "das löst alles" flow: dump an interesting
  anchor → reload + `runtime_trace_start` (746.2) → replay-with-trace records the
  fine firehose from there.
- **No scrub UI** on the Live tab for now (user: "der Buffer darf im Hintergrund
  schreiben, den rest sehen wir dann"). The recorder runs headless; the API is
  there for when the UI is built.

## 7. Build slices (feature-complete; each ships with a gate)

1. **766.1 — SAB handoff ring + record framing.** The small `SharedArrayBuffer`
   handoff + `Atomics` cursors + frame format + the loop's `deliver()`
   (memcpy-only). Gate: a probe that fills + drains across two threads, asserts
   byte-exact handoff + overwrite-oldest on backlog + zero allocation on the
   producer (alloc counter).
2. **766.2 — anchor producer on the tick loop.** Replace the inline ring capture
   with `deliver(ANCHOR)` (flat-pack scalars + RAM memcpy, no framebuffer). Gate:
   probe-705b equivalent — worker reconstructs a byte-exact anchor; capture path
   does zero per-frame allocation.
3. **766.3 — medium gen-gate, BOTH media.** `diskWriteGeneration()` in the facade
   (Spec 612 PL-5) + cart `writableGeneration()`; loop delivers medium only on
   change. Gate: read-only disk/cart = medium delivered once; a write bumps gen →
   delivered again; restore rolls back correctly (the 761 4b case, both media).
4. **766.4 — worker: store + dedup + index.** Full anchors round-robin in the
   ~32 MiB store (no diff), medium deduped by generation, cycle→offset index.
   Gate: ~3-4 min depth at 0.5 s; reconstruct any stored anchor byte-exact;
   medium stored once per version; pure overwrite-oldest (no pin, RQ4).
5. **766.5 — worker API + re-point scrub/dump + 3-gate dump.** WS `checkpoint/*` +
   MCP `runtime_checkpoint_*` + the `.c64re` dump-from-anchor (1-frame repaint) onto
   the worker; drop pin (RQ4). The explicit dump reachable from all three gates
   (Monitor command / MCP tool / UI button). Gate: 707 dump/undump green via the
   worker; e2e-761 scrub/resume green (pin-less: resume = restore+continue, stale
   future rolls out naturally); restore repaints; dump fires from each gate.
6. **766.6 — perf acceptance.** Recorder ON by default; daemon holds ~50 fps +
   audio underrun-free 60 s on the live UI (user ear) — EF AND disk. The bar
   BUG-049 set.

## 8. Gates (must stay green throughout)

- `probe-705b` (capture→restore byte-exact), `probe-707` (.c64re), `e2e-761`
  (scrub/resume), `probe-single-path`, the 7-game screenshot gate (restore must

- New: `probe-766-ring` (two-thread byte-exact handoff + zero producer alloc),

- Perf (§7.6): user ear on the live session — the BUG-049 acceptance bar.

## 9. Open refinement questions (decide WITH the user before the affected slice)

- **RQ1 — DECIDED (2026-06-15): TWO buffers (b).** A small `SharedArrayBuffer`
  handoff ring (loop→worker, raw frames, round-robin) + a separate worker-owned
  128 MiB compacted store (a normal `ArrayBuffer` — only the worker + the UI API
  touch it, and the UI gets data over WS, so it need NOT be shared). Each buffer
  has ONE job: the producer never touches the compacted store; the worker never
  fights the producer over a layout. Costs ~a few MiB extra (the handoff) — a
  non-issue on the 24 GB target. (a) one shared 128 MiB block was rejected: the
  loop's raw frames and the worker's compacted deltas would share one layout =
  fragmentation + bookkeeping mess. Handoff size = RQ2-dependent (a few seconds of
  raw anchors); sized in 766.1.
- **RQ2 — DECIDED (2026-06-15): 0.5 s cadence.** One 64 KiB memcpy 2×/s on the
  loop = microseconds. (Finer is cheap if ever wanted.)
- **RQ3 — DECIDED (2026-06-15): full anchors, NO delta encoding.** 2-3 min scrub is
  ample; a ~32 MiB store at 0.5 s holds ~3-4 min of full anchors. The worker is a
  dumb-fast copier+indexer + medium-deduper — no diff, no delta format, no CPU
  burned. (Delta-diffing "smelled like a CPU killer" and bought history we don't
  need.) Medium dedup is still required (else +1 MiB/anchor).
- **RQ4 — DECIDED (2026-06-15): NO pin. Pure overwrite-oldest** in both the handoff
  and the store (the rolling ~3-4 min window). Nothing automatic keeps an anchor
  alive. Durability is ONLY the **explicit dump** to `.c64re`, reachable from three
  gates (Monitor command / MCP tool / UI button) — never automatic. 705.B/761's
  pin + resume-auto-pin are dropped when scrub/resume re-points onto the worker
  (resume = restore + continue; the stale future just rolls out of the window).

## 10. Planned follow-up — the TRACE rides the same shared memory (user, 2026-06-15)

Once the recorder's shared-memory firehose stands, the binary trace (726.B)
should be rebuilt to consume the SAME stream — one cheap producer in the main
loop, multiple consumers:

```
MainLoop ──► shared-memory firehose ◄── RingBuffer worker (always — scrub anchors)
                                    ◄── Trace store (on command)
```

Today the trace is a SECOND producer on the emu thread with its own
drain/backpressure — exactly the coupling that pulls fps. Folding it into the
one shared-memory stream removes that.

**The tension to resolve in that follow-up spec (NOT here):** the recorder is
**lossy** (fps-first — an overrun is a benign history gap), but the trace is
**no-drop** (726.B: the recorded CPU stream is the truth anchor for port-debug).
A lossy ring would drop cpu_steps → trace gaps. Resolvable — e.g. trace-on
enables backpressure/no-drop on the shared stream (the user opted into the trace,
so the fps cost is a fair, explicit trade) — but it is a real design decision.

Sequencing: build 766 (recorder) fully first (it fixes BUG-049); the trace
unification is its own complete spec after. The trace works today (726.B); this
is unification, not a missing feature.

## 11. Non-goals

- NOT moving the CPU firehose / DuckDB onto the hot path (726.B stays; the worker
  here is the checkpoint/anchor worker).
- NOT a scrub UI yet (headless recorder + API only).
- NOT changing the `.c64re` on-disk format (707 stays; the worker produces it).
- NOT a second runtime/process (the daemon + one product UI remain the surfaces).
