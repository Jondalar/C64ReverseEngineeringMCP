# Spec 447.5.x — Pawn `LOAD"$",8` FILE NOT FOUND root-cause (code-audit-first)

**Status:** OPEN
**Priority:** HIGH (V1-ship gate dependency — Pawn was original
1541-literal-vice motivation, [[feedback_drive_harder]])
**Parent:** Spec 447.5 PARTIAL
**Doctrine:** Code-audit-first per [[feedback_read_vice_first]].
This is KERNAL standard serial-bus protocol, NOT M-W fastloader
bit-bang. The bug MUST be readable in code — VICE handles the
same standard path against the same drive ROM bytes; TS-side
divergence is in glue / wiring, not in the 1:1-audited 6502
microcode.

## Pawn smoke result (post-Spec-447.5)

Both `samples/the_pawn_s1.g64` and `samples/the_pawn_s2.g64`:
- mount succeeds (Fix #6 unblock)
- C64 boot to BASIC READY ✅
- typeText `LOAD"$",8\r` reaches BASIC
- Screen shows: `SEARCHING FOR . / .FILE NOT FOUND ERROR / READY.`
- C64 PC ends at `$e5cf` (BASIC error handler)
- Drive PC ends at `$ec3d` / `$ec71` (drive ROM standard idle / error)

Spec 447.5 fixes (empty-track 0x55, preload-168, zones port,
graceful mount) were necessary but **not sufficient**. Standard
KERNAL `OPEN 0,8,0,"$"` workflow still fails.

## Why this MUST be code-visible

KERNAL LOAD"$",8 path is well-trodden VICE territory:
1. C64 KERNAL LOAD ($F49E) → SETLFS → SETNAM → CHKIN-style
   serial flow.
2. CIA2 PA serial bit-bang (ATN/CLK/DATA pulses).
3. IEC bus delivers LISTEN ($28+8) + SECOND ($F0+0 channel) +
   filename "$" + UNLISTEN.
4. Drive ROM main loop ($EAA0+) sees ATN, receives bytes, parses
   filename, recognizes "$" = directory request.
5. Drive ROM allocates DOS directory buffer, sends back BAM
   header + dir entries via TALK + READ.
6. C64 KERNAL LOAD reads bytes, stores at $0801+, returns READY.

Every step is in 1:1-ported component. M-W bit-bang fastloader
foo is NOT involved. The bug is in TS-side composition — IEC
bus glue, drive serial-receive wiring, or a timing window we
broke.

## Code-audit plan (read VICE first, then TS)

Stop at first divergence. Do not skip to bilateral trace.

### Step 1 — Drive ROM final PC

End state shows drive PC = `$ec3d` (s1) or `$ec71` (s2). These
are inside the 1541 stock ROM `$EAA0 + idle loop` region.
**Look up exactly what these addresses correspond to** in the
1541 ROM disassembly. If they're "waiting for ATN" or
"command-parsing-error", that narrows the divergence point.

### Step 2 — IEC bus traffic during LOAD"$",8

Audit `IecBus` (`src/runtime/headless/iec-bus.ts`) instrumentation
in true-drive mode. Count LISTEN frames, second-addr frames, byte
frames, UNLISTEN frames seen by drive side during the 100M-cycle
window. Compare to expected protocol shape:
- exactly 1 LISTEN($28+8)
- exactly 1 SECOND($F0+0) (= OPEN, channel 0)
- 1 byte "$" (0x24) under listen
- 1 UNLISTEN($3F)
- (then C64 sends TALK + reads data)

If TS sees zero LISTEN frames or only partial sequence → bug
upstream in CIA2 serial bit-bang. If sees full LISTEN but no
TALK follow-through → bug in DOS directory-channel state.

### Step 3 — C64 KERNAL real-execution

Confirm C64 CPU executes the KERNAL serial path (`$ED40`-ish
range) during the LOAD sequence. We already verified
`enableKernalSerialTraps: false` in true-drive mode → traps OFF.
But verify no other shortcut exists (e.g. hidden CIA2 / serial
shim).

### Step 4 — Drive serial-receive wiring

Audit `Via1d1541` ↔ IEC bus glue (the drive's IEC interface):
- VIA1 PB drives DATA/CLK/ATN lines out
- VIA1 PA reads ATN-IN, CLK-IN, DATA-IN
- Drive ROM polls PB for ATN edges (`bit $1800; bvc loop`)
- onATN routine fires, drive enters serial-receive mode

Compare 1:1 vs VICE `viad1541.c` PA/PB routing.

### Step 5 — DOS directory-channel state

Drive ROM at receive completion parses filename. For "$" it
should set DOS state for directory-mode. If filename buffer ends
up empty / corrupted → DOS replies with FILE NOT FOUND (`$62`).

Audit drive RAM around DOS state variables ($0200-$02FF buffer
area) at end of LOAD attempt. If filename byte is wrong, the
serial-receive wiring is the culprit.

### Step 6 (FALLBACK ONLY if 1-5 stall) — bilateral DuckDB trace

Per [[feedback_trace_into_duckdb]]: TS + VICE side-by-side trace
of cpu_step + iec_edge + drive_step + via1_pb_read for an
identical workflow window. DuckDB-diff first divergence row.

**Forbidden as Step 1.** This is standard protocol; reading code
MUST surface the bug.

## Do Not

- **Don't open trace capture as Step 1.** Read the code first.
- **Don't add TS-EXTRA shortcuts.** If audit surfaces hidden trap
  / fallback, delete the shortcut, don't paper over.
- **Don't break canary:spec-430.** Each fix gets re-run sanity.
- **Don't refactor unrelated layers.** Scope is Pawn LOAD"$",8
  read path. M-W bit-bang fastloaders are out of scope (separate
  audit if needed).

## Out of scope

- Custom protocol fastloaders (Krill, EXOS, BURST, etc.) — these
  use M-W + custom bit-bang and have their own debug paths.
- D64 image parsing — Pawn ships G64, D64 is separate.
- Pawn copy protection logic (track 18 custom encoding, if any) —
  if such encoding is present, that's its own follow-up; for
  LOAD"$",8 the standard directory must work first.
