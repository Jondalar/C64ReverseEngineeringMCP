# Spec 450.x — RED_OK SAVE/FORMAT root-cause (code-audit-first)

**Status:** OPEN
**Priority:** HIGH (V1-ship gate flip dependency)
**Parent:** Spec 450 (validation harness DONE-with-CAVEATS)
**Doctrine:** **Code-audit-first.** No trace capture as the
starting move ([[feedback_read_vice_first]]). VICE components in
TS are 1:1-audited per Specs 441-448; divergence must be in
TS-side glue / composition / hidden shortcut. Read the code,
find it, fix it.

## Mandate

Flip 3 RED_OK rows from Spec 450 (B5 format-prg, C8
basic-save-load, C9 g64-extended-tracks-save) to PASS by finding
the single root-cause for "drive trackBuffer never engages on
KERNAL SAVE/FORMAT" and patching the responsible TS-side wiring.

## Why not trace-capture first

Per 1541-only mandate ([[feedback_vice_no_alternatives]]) +
1541 port workflow ([[feedback_1541_port_workflow]]):

- KERNAL ROM, drive ROM, IEC bus, VIA1, VIA2, GCR, alarm,
  drivecpu — ALL audited 1:1 against VICE.
- If write doesn't happen, the cause is NOT in the 1:1 code —
  it's in TS-side glue, composition, or an unexamined shortcut.
- Both are findable by **reading the code.** Capture only helps
  when reading-the-code stalls.

[[feedback_trace_into_duckdb]] requires DuckDB-trace for
divergence localisation — that doctrine applies as fallback, NOT
as first move when code-audit can answer faster.

## Code-audit plan (stop at first divergence)

### Step 1 — `writeProtected` default audit

- `IntegratedSessionOptions.writeProtected`: confirm default
  (probably undefined → false, but verify).
- Trace propagation: opts.writeProtected → kernel → drive →
  VIA2 PB_WPS pin → drive ROM check.
- Our 450 scenarios pass `writeProtected: undefined` to
  `runIntegratedScenario`. Verify this resolves to writable on
  the drive side.

### Step 2 — VIA2/GCR write-back chain

- `src/runtime/headless/drive/via2-gcr.ts` — drive write nibble
  output.
- `src/runtime/headless/drive/via2-gcr-shifter-coupling.ts` —
  shifter buffers bytes from VIA2.
- `src/runtime/headless/drive/head-position.ts` `TrackBuffer` —
  where modifications should land + `.isModified()` returns true.
- Find the wire that connects shifter writes → trackBuffer dirty
  marking. Verify it exists AND is hooked up in true-drive mode.

### Step 3 — IEC bus LISTEN+OPEN+second-addr

- `src/runtime/headless/iec-bus.ts` + drive-side IEC handler.
- C64 KERNAL sends:
  - ATN + LISTEN ($28+device) + second-addr ($F0+channel)
  - bytes (filename for OPEN, payload for SAVE)
  - UNLISTEN
- Drive ROM at IEC routine should receive these. Verify drive
  ROM IS receiving the LISTEN frame for SAVE (channel 0) /
  command channel (channel 15).
- If drive ROM doesn't see LISTEN, the C64-side serial bit-bang
  is broken (despite B6 passing — diskid is a SYS-direct test
  that bypasses KERNAL channel I/O).

### Step 4 — KERNAL SAVE routine real-execution

- KERNAL SAVE at $FFD8 dispatches to $F5ED.
- $F5ED → SETLFS / SETNAM context → CHKOUT to channel → loop
  CHROUT each byte → CLRCHN.
- Confirm CPU executes this path (not a TS-EXTRA trap we
  missed in Step 1).
- Quick test: instrument c64Cpu with a breakpoint at $F5ED;
  scenario B5 should hit it during SAVE workflow.

### Step 5 (fallback) — bilateral DuckDB capture

ONLY if Steps 1-4 all confirm code is correct and bug still
invisible. Per [[feedback_trace_into_duckdb]]:

- TS: `vice_trace_runtime_start` against integrated session,
  capture cpu_step + iec_edge + drive_step for SAVE workflow.
- VICE: same workflow on real VICE, capture same trace families.
- Load both into trace store, DuckDB-diff first divergence row.

## Acceptance

1. Root cause identified + fixed.
2. B5 / C8 / C9 flip from RED_OK to PASS in
   `canary:spec-450-write`.
3. Bilateral VICE baselines captured under
   `samples/baselines/spec-450-write/{B5,C8,C9}/` (now that
   write workflow actually produces a different image — capture
   becomes meaningful).
4. Image-compare assertion enabled (drop
   `selfConsistencyOnly: true` for these scenarios).
5. Production-proof updated: V1-ship verdict flips from PARTIAL
   to GREEN.

## Do Not

- **Don't open a trace capture as the first step.** Read the
  code first. The 1:1-audit doctrine guarantees the bug is in
  TS-side glue, not in component ports.
- **Don't add TS-EXTRA workarounds.** If the audit surfaces a
  TS-side shortcut (e.g. a forgotten trap, a hardcoded
  fallback), delete the shortcut, don't paper over.
- **Don't break the existing 6 PASS scenarios.** Re-run full
  `canary:spec-450-write` after each fix.
- **Don't capture massive traces "to be safe".** If trace IS
  needed (Step 5 fallback), scope it tightly: SAVE workflow
  window only, cpu_step + iec_edge + drive_step (no mem_read /
  mem_write spam).

## Risk

Medium. Code-audit through 4 steps is bounded effort. Most
likely outcome: a small TS-EXTRA glue issue (e.g. drive write
events not propagating to TrackBuffer, or a missing wire from
shifter to track buffer). Fix is a 1-3 line patch + scenario
re-run.

If Step 5 fallback is reached, scope is larger (bilateral trace
work) but still bounded — same methodology that proved Sprint
430 motm AB fastloader root cause.

## Out of scope

- DOS command channel full coverage (channel-15 status reads,
  dir-listing) — post-V1.
- 1571 / 1581 / parallel-cable SAVE — post-V1.
- NTSC SAVE — Spec 451 follow-on.
