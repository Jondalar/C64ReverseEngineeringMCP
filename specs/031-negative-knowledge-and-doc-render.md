# Spec 031: Negative Knowledge And Markdown Doc Render

## Problem

Findings and open questions capture what may be true. Real work also
needs explicit "do not try this again" knowledge: refuted theories,
failed heuristics, dangerous commands, stale assumptions, and
project-specific rules ("do not trust memory system for Bug 5; read
trace first"). Today this lives in `docs/ANTI_PATTERNS.md` written
by hand. Separately, Sprint 18 (knowledge tabs) exposes findings /
entities in the UI but does not render Markdown for external sharing
or git review. REQUIREMENTS R21 plus the deferred R10 markdown
renderer.

## Goal

Two related capabilities, one sprint:

1. **Negative knowledge** is a first-class status on findings and a
   dedicated record kind for project-level anti-patterns.
2. A doc renderer keeps `docs/FINDINGS.md`, `docs/ENTITIES.md`,
   `docs/ANTI_PATTERNS.md`, and `docs/OPEN_QUESTIONS.md` in sync
   with the knowledge JSON so external readers / git diffs see the
   same content the UI tabs show.

## Approach

### Negative knowledge

Extend `FindingRecord.status` enum:

```ts
status: "draft" | "active" | "confirmed" | "refuted" | "stale" | "dangerous";
```

New optional fields:

```ts
refutationEvidence?: EvidenceRef[];
replacedBy?: string;            // id of the finding that supersedes
```

New top-level record kind `AntiPattern`:

```ts
interface AntiPattern {
  id: string;
  title: string;
  reason: string;
  severity: "info" | "warn" | "error";
  evidence?: EvidenceRef[];
  appliesTo?: { phase?: string; toolName?: string; commandPattern?: string };
}
```

Stored at `knowledge/anti-patterns.json`.

MCP tools:

- `save_anti_pattern(...)` / `list_anti_patterns(...)`.
- `mark_finding_refuted(finding_id, refutation_evidence)`.
- `mark_finding_dangerous(finding_id, reason)`.

### Onboarding integration

`agent_onboard` lists high-severity anti-patterns and recently
refuted findings before tool suggestions. `agent_propose_next`
filters out actions that match a registered anti-pattern's
`commandPattern` unless the proposing context cites new evidence.

### Markdown doc render

New library `src/doc-render/`:

| Source store | Generated doc |
|--------------|---------------|
| `findings/*.json` | `docs/FINDINGS.md` (grouped by tag/topic) |
| `entities/*.json` | `docs/ENTITIES.md` (grouped by kind) |
| `open-questions/*.json` | `docs/OPEN_QUESTIONS.md` (grouped by priority) |
| `flows/*.json` | `docs/FLOWS.md` |
| `anti-patterns.json` | `docs/ANTI_PATTERNS.md` |
| `project-profile.json` | `PROJECT_PROFILE.md` |
| `views/memory-map.json` | `docs/MEMORY_MAP.md` (optional) |

MCP tool `render_docs(scope?: "all" | "findings" | "entities" | ...)`.

Triggers:

- Single `save_finding` / `save_entity` / `save_open_question` /
  `save_anti_pattern` / `save_project_profile` calls re-render the
  matching doc in-band by default.
- Bulk operations (`bulk_import_analysis_reports`,
  `register_existing_files`, `project_repair`, etc.) accept a
  `defer_render: boolean` arg (default `true` for bulk). With
  `defer_render` set, individual saves skip the per-call render
  and a single `render_docs("all")` runs once at the end.
- `build_all_views` always re-renders everything.

Generated files carry a header:

```markdown
<!-- generated by c64re render_docs; do not edit by hand -->
```

Hand-edited Markdown without that marker is left alone.

## Acceptance Criteria

- A finding can be marked `refuted` with evidence; the prior
  finding stays visible with its replacement linked.
- A registered anti-pattern with `commandPattern: "rm -rf"` causes
  `agent_propose_next` to omit any action whose command matches.
- `render_docs` produces `FINDINGS.md`, `ENTITIES.md`,
  `OPEN_QUESTIONS.md`, `ANTI_PATTERNS.md`, `PROJECT_PROFILE.md` on
  the BWC project; re-running is idempotent.

## Tests

- Smoke: render all docs on the fixture project; assert files
  exist and round-trip is stable.
- Smoke: register an anti-pattern, assert filter behavior in
  `agent_propose_next`.

## Out Of Scope

- Diffing two doc renders.
- Custom Markdown templates.

## Dependencies

- Sprint 18 (knowledge tabs) for UI surface of new statuses.
- Sprint 23 (Spec 026 project profile) for `PROJECT_PROFILE.md`
  source.
