# Spec 444.x — drivecpu.c literal-shape WITH caller-contract audit

**Status:** OPEN
**Priority:** MEDIUM (cycle-diff regression ±6 documented; not
LOAD-blocking after hotfix revert 80af949, but doctrine-gap)
**Parent:** Epic 440
**Depends on:** Spec 444 v2 HOTFIX REVERT (commit `80af949`)

## Why

Spec 444 v2 (commit `9e2edd8`) ported `drivecpu.c` to literal VICE
shape with 4 bundled review-fix changes:

1. `executeToClock` loop: stop_clk-based (additive across calls),
   10000-cycle outer chunking, `cycleAccum &= 0xffff` fractional
2. `softReset` preserves `cycleAccum` + leaves `lastClk` unchanged
3. `drivecpuWakeUp` stale-clock-skip (16M cycle threshold)
4. VIA2 IRQ chip-push (`attachIrqLine`, no boundary polling)

These changes achieved 9999/9999 ±2 cycle-diff vs VICE im2 baseline
(vs 8805/9999 ±6 in v1 math-equivalent form). **BUT broke
LOAD"*",8,1 on ALL disks** — drive RAM cmd buffer never populated,
drive PC stuck at KERNAL READY ($e5cd region) after LOAD.

Bisect 2026-05-15 narrowed to single commit `9e2edd8`. Hotfix
revert in `80af949` restored cd9ad49 state for `drive-cpu.ts` +
`via2d1541.ts` only.

## Root-cause hypothesis (untested)

TS caller contracts for `softReset` and/or `executeToClock` and/or
VIA2 attach-order assume v1 math-equivalent semantics. Literal-VICE
shape requires TS-caller-side migration that the v2 commit did
not perform.

Tests 2A (VIA2 polling restored alone) and 2C (executeToClock loop
reverted alone) BOTH still RED in bisect — meaning multiple
sub-changes combine, OR single sub-change has masking dependency
on other sub-changes.

## Investigation plan

### Phase 1 — Isolate culprit(s)

Cherry-pick reversal on top of 9e2edd8 (one at a time):

- **A.** softReset cycleAccum-clear restored (v1 behaviour)
- **B.** VIA2 polling-at-boundary restored, chip-push commented
- **C.** executeToClock cycleAccum-loop restored, stop_clk-additive dropped
- **D.** drivecpuWakeUp call commented out

For each: build + `node scripts/test-mm-screenshots.mjs`.
The reverted-change that restores GREEN identifies the culprit
(may be more than one).

**Already tested 2026-05-15 (hotfix path):**
- 2A (B reverted alone) — STILL RED
- 2C (C reverted alone) — STILL RED
- 2(A+C) combined = full file revert — GREEN

So **B + C both load-bearing for the bug**. A and D not yet
isolated.

### Phase 2 — Caller-contract audit for culprit(s)

For each identified change, grep ALL callers of affected API:
- **softReset (A):** integrated-session, kernel cold-reset, tests
- **VIA2 setInt (B):** backend wiring, attach-order in DriveCpu ctor
- **executeToClock (C):** scheduler cycle-wrappers, IEC bus events
- **wakeUp (D):** trigger paths (disabled→enabled, mid-run sync)

For each callsite, ask: "Does it satisfy the VICE-equivalent caller
contract this literal-shape requires?"

E.g. VICE `softReset` is called from `interrupt_trigger_reset`
asynchronously at instruction boundary, after which `last_clk` is
implicitly `maincpu_clk`. TS callers calling `softReset()` inline
need to `setSyncBaseline` immediately after, OR use the c64Clk-
argument signature.

### Phase 3 — Apply caller migration

Update each callsite to satisfy literal contract. NOT by reverting
the literal shape — by porting the caller side properly.

### Phase 4 — Verify both gates green

- `node scripts/test-mm-screenshots.mjs`: GREEN (PC ∈ game region)
- `node scripts/test-motm-screenshots.mjs`: GREEN
- `node scripts/test-scramble-screenshots.mjs`: GREEN
- Spec 444 cycle-diff: 9999/9999 ±1 or ±2
- `canary:spec-430`: 5/5 PASS

If gates cannot BOTH be green simultaneously, literal-shape may be
incompatible with TS scheduling model. **STOP-and-ask user.**
- Option A: keep math-equivalent (current cd9ad49 baseline) as
  DEVIATION-DOCUMENTED.
- Option B: deeper TS-scheduler port to match VICE alarm-driven model.
