## Why this spec exists

`src/disk/g64-parser.ts` was NEVER literal-VICE-audited. Epic 440
specs 441-449 covered drive chips + emulation primitives + GCR
helpers, but disk-file → track-buffer parsing fell in the gap.

Concrete evidence: **The Pawn s1 cannot load directory** (`LOAD"$",8`
→ `?FILE NOT FOUND ERROR`) in current runtime while VICE loads it
fine. Review-session 2026-05-15 identified at least 4 divergences
between TS `g64-parser.ts` and VICE `src/diskimage/fsimage-gcr.c`
+ `src/diskimage/diskimage.c`. Most load-bearing: empty-half-track
handling returns `null` in TS vs 0x55-filled canonical-size buffer
in VICE. Pawn G64 likely has sparse half-track TOC entries
(offset=0) that drive ROM steps through during seek/error-recovery.

Pawn was the original motivation for the `1541-literal-vice` rewrite
(session 2026-05-13 19:11Z, branch creation). The original bug
diagnosis was in `inspectGCRTrack` (byte-aligned scan + pair-parity
fixed-gap heuristic). That bug-class was fixed in Spec 437/445 for
the inspection-tier API but `g64-parser.ts` was never re-audited
for production-tier divergences.

## Required ports (the actual fixes)

### Fix #1 — Empty-track handling (LOAD-BEARING for Pawn)

VICE `fsimage_gcr_read_half_track` lines 174-178:
```c
} else {  // track-offset == 0
    raw->size = disk_image_raw_track_size(image->type, half_track / 2);
    raw->data = lib_malloc(raw->size);
    memset(raw->data, 0x55, raw->size);
}
```

TS `g64-parser.ts:264-276` currently:
```ts
const offset = this.trackOffsets[slotIndex];
if (offset === 0) {
    return null;     // ← divergence
}
```

**Port:** Return canonical-zone-sized buffer filled with 0x55 when
offset == 0:
```ts
if (offset === 0) {
    const halfTrack = this.slotIndexToHalfTrack(slotIndex);
    const size = disk_image_raw_track_size("g64", halfTrack >> 1);
    const buf = new Uint8Array(size);
    buf.fill(0x55);
    return buf;
}
```

Audit `rotation.ts` null-guards (lines 320, 343, 946, 959) — after
this fix `GCR_track_start_ptr` should always be non-null for mounted
disks. The null branches may stay as defense but should not be the
hot path.

### Fix #2 — `disk_image_raw_track_size` + `disk_image_speed_map` port

New TS module (suggest `src/disk/disk-image-zones.ts`) or extend
existing `gcr.ts`:

```ts
// VICE diskimage.c:201-207
const RAW_TRACK_SIZE_D64 = [6250, 6666, 7142, 7692];

// VICE diskimage.c:82-93 (1541 G64/D64/P64/D71/G71 branch)
export function disk_image_speed_map_g64(track: number): number {
    return (track < 31 ? 1 : 0)
         + (track < 25 ? 1 : 0)
         + (track < 18 ? 1 : 0);
}

// VICE diskimage.c:241-264
export function disk_image_raw_track_size(format: "g64", track: number): number {
    return RAW_TRACK_SIZE_D64[disk_image_speed_map_g64(track)];
}
```

Test: hand-pinned values for tracks 1, 17, 18, 24, 25, 30, 31, 35.

### Fix #3 — Track-length validation

VICE `fsimage_gcr_read_half_track` lines 154-159:
```c
track_len = util_le_buf_to_word(buf);
if ((track_len < 1) || (track_len > max_track_length)) {
    log_error(..., "Track field length %u is not supported.", track_len);
    return -1;
}
```

TS only checks file-bounds. Add `track_len <= maxTrackSize` check.
Throw on violation (= VICE log_error + return -1; TS = throw new
Error since mount is one-shot).

### Fix #4 — Pre-load semantics audit

VICE `fsimage_read_gcr_image` loops all 168 half-tracks at mount time.
TS lazy-loads via `ensureTrack`. Decide:

- **Option A (literal):** pre-load all half-tracks at parser
  construction. Memory cost = up to 168 × 7692 ≈ 1.3 MB per disk.
  Truly 1:1.
- **Option B (DEVIATION-DOCUMENTED):** keep lazy-load but ensure
  null-fallback in `ensureTrack` returns canonical 0x55 buffer
  per fix #1. Functionally equivalent for drive emulation.

User decision required per Spec 440 step 7 (no arch w/o ask).
**Recommendation: Option B** — lazy + null-fallback to 0x55 buffer
produces identical drive-visible behaviour without the up-front
memory cost. Mark explicitly as DEVIATION-DOCUMENTED in mapping.

### Fix #5 — `inspectGCRTrack` migration (cleanup)

`src/disk/gcr.ts:1012` `inspectGCRTrack` uses pair-parity-heuristic
+ fixed header→data pairing (the original Pawn bug-class from
2026-05-13). Now lives only in inspection-tier `G64Parser.inspectTrackBlocks`
(g64-parser.ts:443). Per Spec 445 Phase 2c-cleanup doctrine:
"inspection-tier API" allowed but should be marked as such.

**Audit:** is `inspectGCRTrack` still called by any production code
path? `grep -rn "inspectGCRTrack" src/ scripts/ tests/`. If not,
keep as inspection-tier with explicit `@internal inspection-tier`
docstring (analog to other Spec 445 legacy variants). If called by
production, migrate to `gcr_find_sector_header_vice` /
`gcr_read_sector_vice`.

## Risk + scope

**Caller-impact small** — `getRawTrackBySlotIndex` return-shape stays
`Uint8Array | null` (null still possible for actual file-corruption
case post-validation, just not for offset==0). Existing callers
treat null as "track not present" so no API break.

**Behaviour-impact possibly significant** — drive emulation now sees
0x55 bytes where it previously saw "no rotation"; SYNC line will
never assert on those bytes (= correct VICE behavior). Drive ROM
will spin in "waiting for sync" instead of hanging on null. Could
unblock other titles besides Pawn (LNR S1 candidate — half-track-rich
G64 might be triggering same null path).

**Regression risk** — minimal. Empty-track was producing null + drive
state-machine hung. After fix it sees 0x55 = consistent garbage =
same as a real but unreadable track. Could be slightly slower (drive
spins instead of erroring) but correctness > perf for emulation
fidelity.
