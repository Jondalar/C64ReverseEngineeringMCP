# Spec 042: Annotation Helper — `propose_annotations`

## Problem

Phase 5 (Semantic Analysis V1) is the most labor-intensive and
ambiguous step. Today the agent reads the entire ASM and writes
`*_annotations.json` from scratch in free-form judgment.
REQUIREMENTS R15.

## Goal

`propose_annotations` runs a deeper second-pass classifier on
`*_analysis.json`, applies a battery of pattern fingerprints, and
emits a draft `*_annotations.draft.json` file. The agent reviews,
edits, and writes the final `*_annotations.json` (or accepts the
draft).

## Approach

### Pattern fingerprint modules

`pipeline/src/analysis/annotators/` contains:

- `kernal-call-fingerprints.ts` — known JSR sequences (SETNAM /
  SETLFS / LOAD trio, CHROUT loop, IRQ vector hook).
- `loop-shape-fingerprints.ts` — for-loops, screen-clears,
  raster-waits (`bit $D011 / bpl ...`), joystick-polls.
- `pointer-table-detector.ts` — runs of aligned 16-bit pairs whose
  high bytes fall in plausible code ranges.
- `string-table-detector.ts` — null-terminated PETSCII series with
  consistent average length.
- `irq-handler-fingerprints.ts` — `pha / txa / pha / tya / pha`
  prologue + matching epilogue + final `rti`.
- `cross-ref-namer.ts` — names a target based on cross-references:
  "called from screen_clear at +N times" or "branch target inside
  same routine".

Each module exports a `runFingerprint(context): AnnotationCandidate[]`
function. The candidate includes confidence, reason, and proposed
annotation type (segment / label / routine / open-question).

### Output schema

`*_annotations.draft.json` mirrors the existing annotations shape
plus per-entry metadata:

```ts
{
  segments: Array<SegmentAnnotation & { confidence: number; reason: string; autoGenerated: true }>,
  labels: Array<LabelAnnotation & { confidence: number; reason: string; autoGenerated: true }>,
  routines: Array<RoutineAnnotation & { confidence: number; reason: string; autoGenerated: true }>,
  openQuestions: Array<{ title: string; description: string; confidence: number; source: "static-analysis" }>,
}
```

The draft is a separate file. Existing `*_annotations.json` is
never touched. Continue mode is option (c) from refinement —
overwrite Draft, manual unchanged.

### Confidence buckets

All candidates emit, regardless of confidence. The CLI / MCP
output summarises by bucket:

- high (≥0.8): N candidates
- medium (0.5-0.8): M candidates
- low (<0.5): K candidates

Agent decides which to accept.

### Optional question persistence

Default: questions land in the draft file only.

With `--persist-questions` (CLI) or `persist_questions: true` (MCP):

- Each `openQuestions[]` entry is also saved via
  `save_open_question(source: "static-analysis", autoResolvable: true)`
  (Spec 036 fields).

### CLI command

`pipeline/cli.cjs propose-annotations <prg.prg> [--analysis <json>]
[--output <draft.json>] [--persist-questions]`.

### MCP tool

`propose_annotations(prg_path?, analysis_json?, output_path?,
persist_questions?)`. Phase tag `[Phase 4-5]`. Output summary +
draft path + bucket counts.

## Acceptance Criteria

- Running `propose_annotations` on the fixture PRG produces a
  `*_annotations.draft.json` with at least one segment, label, or
  routine candidate.
- Re-running overwrites the draft, leaves any existing manual
  `*_annotations.json` untouched.
- `--persist-questions` writes `openQuestions[]` to both the draft
  and the project knowledge layer with `source: "static-analysis"`.
- The CLI prints bucket counts (high/medium/low).

## Tests

- Smoke: run propose_annotations on the HELLO fixture; assert
  draft file exists with auto-generated entries.
- Smoke: assert continue-mode (existing manual file present)
  leaves manual file untouched.
- Smoke: --persist-questions writes the right source tag.

## Out Of Scope

- Auto-merging draft into final annotations file (manual review
  first).
- LLM-driven semantic classification (deterministic fingerprints
  only in v1).

## Dependencies

- Spec 020 platform marker — KERNAL fingerprints come from the
  platform table.
- Spec 036 open-question source tagging — `static-analysis`
  source.
- Spec 034 phase model — phase 4-5 tagging.
